Security — 62 / 100

Two critical authorization gaps and several medium-severity issues. Firebase SDK prevents injection, but client-side admin checks are insufficient.

CriticalAdmin repository functions lack server-side authorization
src/features/admin/data/adminRepository.ts — 12+ exported functions
Functions like createContentJob(), retryJob(), cancelJob(), and approveSubjectPlan() check authentication via getCurrentUserId() but never validate admin role. Any authenticated user can call these directly via the Firestore SDK, completely bypassing the UI-level admin check in AdminLayout. Firestore Security Rules must enforce admin authorization server-side.
CriticalHardcoded local dev endpoint with trace UUID
src/core/providers/contexts/auth/helpers.ts — line 32
Hardcoded http://127.0.0.1:7242/ingest/abd8d170-... endpoint and UUID visible in compiled code. Should be moved to environment variables and gated behind __DEV__.
HighFirestore fallback grants admin access
src/features/admin/hooks/useAdminAuth.ts — lines 64–73
When Firebase custom claims are missing, a fallback reads the Firestore user document's role field. If Firestore rules permit client writes, a user could set their own role to "admin" and gain access. Custom JWT claims should be the only authoritative source.
MediumAccount deletion is non-atomic
src/features/profile/data/profileRepository.ts — lines 464–503
Sequential deletes across 6 Firestore collections without transactions. Partial failure leaves orphaned data. Use batch writes or transactions.
MediumNo input validation on email/password changes
src/core/providers/contexts/auth/actions/accountActions.ts — lines 113, 126
No client-side format validation before Firebase calls. Invalid or malicious emails could be set.
MediumDebug logging captures user IDs to external endpoint
src/core/providers/contexts/auth/helpers.ts — lines 51–71
Auth debug logs include userId and are sent to the local dev server. If that server runs on shared infra, UIDs could leak.

Performance — 68 / 100

Unbounded Firestore subscriptions and missing cleanup guards are the primary concerns. The admin feature creates excessive real-time listeners per component.

CriticalUnbounded collection scan on worker_status
src/features/admin/data/adminRepository.ts — line 784
onSnapshot(collection(db, 'worker_status'), ...) subscribes to the entire collection with no limit or filter. Every worker update triggers a full snapshot. Add query filters or limit().
HighJobCard creates 2 Firestore listeners per card
src/features/admin/components/JobCard.tsx — lines 100–136
Each JobCard subscribes to both factory_jobs and factory_job_runs. A list of 50 jobs creates 100 active real-time listeners. Batch-fetch or consolidate into a single subscription.
HighMissing isMounted guards on 4+ onSnapshot listeners
src/features/admin/data/adminRepository.ts — lines 784, 816, 851, 889
Real-time subscriptions return callbacks without checking if the component is still mounted. Can cause React state-update warnings and memory leaks on rapid navigation.
MediumgetTodayQuote fetches ALL quotes as fallback
src/features/home/data/homeRepository.ts — line 96
If no quote is scheduled for today, getDocs(quotesCollection) fetches the entire collection to pick one at random. Add limit(100) or use document-ID sharding.
MediumThemeContext returns null during AsyncStorage load
src/core/providers/contexts/ThemeContext.tsx — lines 176–178
Provider returns null until theme preference loads, unmounting the entire app tree and causing a blank flash on cold start. Render with a default theme while loading instead.
MediumUnbounded child jobs subscription
src/features/admin/data/adminRepository.ts — lines 667–689
Real-time subscription to all child jobs of a parent with no limit. A subject with 500 course children subscribes to all 500 documents.

Correctness — 72 / 100

Missing return types on async functions, untyped error handling, and production console.log statements are the main concerns. Core logic is sound.

HighMissing Promise return types on 10+ async functions
src/core/providers/contexts/auth/actions/ — credentialActions.ts, sessionActions.ts, accountActions.ts
Async functions like deleteAccount, linkAnonymousAccount, signUp, signIn, etc. lack explicit Promise<void> return type annotations. TypeScript cannot catch missing await at call sites.
HighUntyped error handling — catch (error: any)
credentialActions.ts, accountActions.ts, AuthContext.web.tsx — 6+ instances
Breaks type safety and hides error information. Replace with catch (error: unknown) and use type guards: error instanceof Error ? error.message : 'Unknown error'.
HighUnsafe as any type assertions for Firestore timestamps
src/features/admin/components/FactoryOverview.tsx — lines 385, 509–513
Multiple as any casts when handling Firestore Timestamps. Create a proper type guard: isFirestoreTimestamp(value).
Medium30+ production console.log statements
AuthContext.tsx, SubscriptionContext.tsx, AppProviders.tsx, profileRepository.ts
Startup logs like console.log('[Startup] AuthContext module loaded') pollute the console, slow startup, and expose user IDs. Gate behind __DEV__ or remove.
MediumUnsafe JSON.parse without separate try/catch
src/services/downloadService.ts — line 131
AsyncStorage data parsed with JSON.parse(data). If data is corrupted, the outer catch handles it — but the error message is generic. Parse errors should be caught and handled specifically.
MediumUnguarded optional chaining before property access
src/features/home/data/homeRepository.ts — lines 387–426
series.chapters?.find() handles undefined chapters, but series.title is assumed to exist. If title is missing from Firestore, output is malformed.

ContentManagerScreen.tsx — lines 155, 215, 223
